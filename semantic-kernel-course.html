What is Semantic Kernel?

Discover what Semantic Kernel does and how it helps you build AI-powered applications

The Problem: AI is Powerful, But Complex

Imagine you want to build an AI assistant that can answer customer questions, look up information in your database, and send emails. You'd need to:

1
Connect to AI models

Figure out how to talk to OpenAI, Azure, or other AI services — each with different APIs

2
Give AI access to tools

Let the AI call your database, send emails, or search the web — but you need to wire all that up yourself

3
Handle conversations

Keep track of what the user said, what the AI said, and maintain context across multiple messages

4
Make it reliable

Handle errors, retry failures, monitor performance — all the production concerns

That's a lot of plumbing before you even start building your actual application!

The Solution: Semantic Kernel

Semantic Kernel is like having a universal adapter for AI. It's a framework (a set of tools and rules) that handles all the complex stuff for you:

🔌

Connect Anywhere

Works with OpenAI, Azure, Google, local models, and more — one interface for all of them

🛠️

Add Tools Easily

Give AI access to your functions with just a few lines of code — it'll figure out when to use them

💬

Build Agents

Create smart assistants with personalities, memory, and the ability to use tools

🏢

Enterprise Ready

Built for production with error handling, monitoring, and security baked in

The Code Behind the Magic

Here's what that simple agent looks like in code. Don't worry if this feels unfamiliar — we'll break it all down in the following modules.

CODE 
class MenuPlugin:
    @kernel_function(description="Get menu specials")
    def get_specials(self) -> str:
        return "Special Soup: Clam Chowder"

agent = ChatCompletionAgent(
    service=AzureChatCompletion(),
    plugins=[MenuPlugin()]
)

The AI doesn't just memorize answers — it learns what tools you give it and figures out when to use them. This is like teaching someone to fish instead of giving them a fish. Once you give the AI a tool, it can use it in countless situations you never explicitly programmed.

Meet the Cast

The main actors in Semantic Kernel and what they do

The Main Characters

Every Semantic Kernel application is like a theater production with different actors playing specific roles. Let's meet the cast:

🎭
Kernel

The stage manager — coordinates everything, keeps track of all the actors, and makes sure they can talk to each other

🤖
Agent

The lead performer — the AI that talks to users, makes decisions, and uses tools to get things done

🔌
Plugins

The supporting cast — collections of tools (functions) that agents can use, like a toolbox of capabilities

🌐
Connectors

The translators — they speak the language of different AI services (OpenAI, Azure, Google, etc.) so your code doesn't have to

💾
Memory

The notebook — stores information between conversations so the AI remembers what you talked about before

How They Work Together

Think of building an AI app like assembling a band. Each member has a role, but they need to coordinate to make music together.

CODE 
from semantic_kernel import Kernel
from semantic_kernel.agents import ChatCompletionAgent

kernel = Kernel()
kernel.add_service(AzureChatCompletion())
kernel.add_plugin(MenuPlugin())

agent = ChatCompletionAgent(
    kernel=kernel,
    instructions="You are helpful"
)

Import the Kernel — the orchestrator that holds everything together

Import the Agent — the AI that will interact with users

(blank line)

Create the Kernel (the stage manager)

Connect to Azure's AI service (the brain power)

Add the menu plugin (tools the AI can use)

(blank line)

Create an agent using the kernel

Give it the kernel that has the AI service and plugins

Tell it to be helpful (its personality)

Agent is ready to chat!

ℹ️
Why the Kernel?

You might wonder why we need a Kernel at all. Think of it like a phone directory: instead of every actor needing to know how to reach every other actor, they just ask the Kernel. This makes your code cleaner and lets you swap out pieces (like changing from OpenAI to Azure) without rewriting everything.

The Kernel's Responsibilities

The Kernel is the behind-the-scenes coordinator. Here's what it handles:

📋

Registry

Keeps a list of all available AI services and plugins, like a phone book

📞

Routing

When the AI wants to call a function, the Kernel finds it and executes it

🔍

Configuration

Stores settings like API keys, model names, and behavior options in one place

🛡️

Safety

Applies filters and checks to ensure the AI behaves correctly and safely

How They Talk

Understanding the data flow between components

The Communication Chain

When you ask an AI agent a question, a lot happens behind the scenes. Let's trace the journey of a single question through the system, like following a letter through the postal service.

📬
The Postal Service Metaphor

Think of Semantic Kernel like a postal system. Your question is a letter. The Kernel is the post office that routes it. The Agent is the mail carrier. Plugins are specialized services (like a translator or researcher) that process parts of the letter. The AI service (OpenAI, Azure) is the brain that reads and writes responses.

The Magic: Function Calling

The secret sauce is called function calling. The AI doesn't just generate text — it can decide to invoke your functions!

The Plugin Ecosystem

How to give your AI superpowers with plugins

What Are Plugins, Really?

A plugin is just a collection of functions that share a common purpose. It's like organizing your kitchen: instead of scattering utensils everywhere, you group them — knives in one drawer, spoons in another.

CODE 
class WeatherPlugin:
    @kernel_function(
        description="Get current weather for a city"
    )
    def get_weather(self, city: str) -> str:
        # Call weather API...
        return f"Weather in {city}: Sunny, 72°F"

    @kernel_function(
        description="Get 5-day forecast for a city"
    )
    def get_forecast(self, city: str) -> str:
        # Call forecast API...
        return "5-day forecast..."
PLAIN ENGLISH

Create a plugin class called WeatherPlugin

Mark this as a function the AI can call...

Describe what it does (AI reads this to decide when to use it)

End of the decorator

Define the function — takes a city name as input

Your code here would call a real weather API

Return the weather data as a string

(blank line)

Mark another function...

Describe this one gets forecasts, not current weather

End decorator

Define the forecast function

Call the forecast API

Return the 5-day forecast data

💡
The Power of Descriptions

Those description strings aren't just comments — they're critical! The AI reads them to decide which function to call. If you write vague descriptions, the AI will guess wrong. If you write clear, specific descriptions, it'll nail it every time.

Three Ways to Create Plugins

Semantic Kernel gives you flexibility in how you build plugins:

🐍

From Python Classes

Write a class, decorate methods with @kernel_function — what we just saw

📁

From Prompt Files

Create .skprompt.txt files with prompts — AI generates text based on templates

🌐

From OpenAPI Specs

Point at a REST API's OpenAPI definition — instant plugin from any web service

Multi-Agent Magic

When one AI isn't enough — orchestrating specialist agents

The Specialist Team Approach

Sometimes you don't want one general-purpose AI — you want a team of specialists. Like a hospital: you don't go to one doctor for everything. You see a cardiologist for your heart, a dermatologist for your skin, etc.

Semantic Kernel lets you create multiple agents, each an expert in one area, and have them collaborate.

Real Example: Customer Support Triage

Imagine a customer support system with three specialist agents:

💰
BillingAgent

Expert in charges, payment methods, billing cycles, fees, and payment failures

💵
RefundAgent

Specialist in refund policies, eligibility, processing, and status updates

🎯
TriageAgent

Router that reads user requests and forwards them to the right specialist

How They Collaborate

Here's the clever part: the TriageAgent treats the other agents as plugins! When you give an agent another agent as a plugin, it can "call" that agent like a function.

CODE 
billing_agent = ChatCompletionAgent(
    name="BillingAgent",
    instructions="You handle billing issues..."
)

refund_agent = ChatCompletionAgent(
    name="RefundAgent",
    instructions="You handle refunds..."
)

triage_agent = ChatCompletionAgent(
    name="TriageAgent",
    instructions="Forward to specialists...",
    plugins=[billing_agent, refund_agent]
)
PLAIN ENGLISH

Create a billing specialist agent

Give it a name (for logging/debugging)

Tell it its job: handle billing questions

Done creating billing agent

(blank line)

Create a refund specialist agent

Name it

Tell it to focus on refunds

Done creating refund agent

(blank line)

Create the triage agent (the router)

Name it

Its job: figure out which specialist to ask

Give it the two specialists AS TOOLS it can call!

Triage agent can now "call" the other agents when needed

💡
Agents as Tools

This is a game-changing pattern: agents can use other agents. It's like delegation in an office — the manager (triage agent) doesn't need to know how to do billing or refunds. It just needs to know who to ask.

Memory Matters

How Semantic Kernel remembers conversations and context

The Problem: Goldfish Memory

AI models like GPT don't naturally remember previous conversations. Every time you ask a question, it's like talking to someone with amnesia. They can answer the current question, but they don't remember what you said 5 minutes ago.

Semantic Kernel solves this with memory systems — ways to store and recall information across conversations.

Two Types of Memory

💬
Chat History

Short-term memory — the current conversation. "You said X, then I said Y, then you said Z." Built-in and automatic with agents.

🗄️
Semantic Memory

Long-term memory — facts and knowledge stored in a database. Uses vector embeddings to find relevant information based on meaning, not exact keywords.

Chat History: The Easy Part

Agents automatically track conversation history using a thread. You don't have to do anything special:

CODE 
thread = ChatHistoryAgentThread()

response1 = await agent.get_response(
    messages="My name is Alice",
    thread=thread
)

response2 = await agent.get_response(
    messages="What's my name?",
    thread=thread  # Same thread!
)
# AI will say "Alice" because it remembers
PLAIN ENGLISH

Create a conversation thread (a container for the chat history)

(blank line)

Send a message to the agent and wait for response...

The message: "My name is Alice"

Use this thread to store the history

Done with first message

(blank line)

Send another message (later in the conversation)...

Ask "What's my name?"

Use the SAME thread — so AI can see previous messages

Done with second message

The AI will remember "Alice" from the first message!

Semantic Memory: The Advanced Part

Semantic memory is for long-term facts. Instead of remembering every word, it stores the meaning and can find relevant information based on what you're asking about.

🔍
The Library Analogy

Think of semantic memory like a library with a smart librarian. You don't need to remember the exact title of a book. You can say "I need something about French cooking" and the librarian knows to bring you Julia Child's cookbook — even though you didn't say "Mastering the Art of French Cooking". That's how embeddings work: they understand meaning, not just keywords.

The Big Picture

Putting it all together — understanding the full architecture

The Complete System

Now that you've met all the pieces, let's see how they fit together in a real application. Here's the full architecture of a production Semantic Kernel app:

👤
Your Application

Your code that creates agents, defines plugins, and handles user input

🎭
Semantic Kernel

The framework that orchestrates everything — routes calls, manages state, applies filters

🤖
AI Services

External LLM providers (OpenAI, Azure, Google, etc.) that generate responses and decide which functions to call

🗄️
Your Data & APIs

Databases, web services, file systems — anything your plugins access to get work done

Why This Architecture Matters

This design gives you three huge advantages:

🔄

Flexibility

Swap OpenAI for Azure or local models — just change the connector, not your whole app

🧩

Composability

Build small, focused plugins and agents, then combine them in powerful ways

🛡️

Safety

Control what AI can access — give it only the tools it needs, nothing more
